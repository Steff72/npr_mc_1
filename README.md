# Retrieval-Augmented Generation for Aviation Manuals

This repository contains a notebook-first RAG project that evaluates retrieval and answer generation over four Edelweiss Air Airbus A340 manuals. The entire workflow lives in [npr_mc_1.ipynb](/Users/stefanbinkert/Documents/FHNW_DS/NPR/NPR_MC_1/npr_mc_1/npr_mc_1.ipynb): document loading, section-aware preprocessing, chunking, embedding, retrieval, reranking, generation, evaluation, plotting, and qualitative error analysis.

## What the project does

The notebook builds and compares a series of RAG configurations for a highly structured, safety-critical document collection:

- `FCTM`: Flight Crew Techniques Manual
- `OM A`: Operations Manual Part A
- `QRH`: Quick Reference Handbook
- `CSPM`: Cabin Safety Procedures Manual

Instead of the original mini-challenge cleantech dataset, this project uses aviation manuals plus a manually curated gold evaluation set of 100 question/answer/context items stored in `data/gold_standard_triplets_100_v2.json`.

## Core workflow

The notebook follows this sequence:

1. Load PDF manuals with `PyPDFLoader`.
2. Infer structured section labels from page text and skip likely table-of-contents pages.
3. Build metadata-aware `Document` objects with `manual`, `section`, and `page`.
4. Load the 100-item gold-standard evaluation set.
5. Evaluate retrievers with graded relevance and `NDCG@4`.
6. Evaluate generators with an LLM judge over correctness, groundedness, and completeness.
7. Compare retrieval design choices:
   - baseline recursive chunking (`chunk_size=1000`, `chunk_overlap=200`)
   - title-based chunking with `unstructured`
   - `text-embedding-3-small` vs `text-embedding-3-large`
   - optional cross-encoder reranking with `BAAI/bge-reranker-base`
8. Compare generator models:
   - `gpt-5`
   - `gpt-5-mini`
   - `gpt-5-nano`
9. Produce tables, plots, and qualitative error analysis.

## Evaluation design

Retriever evaluation is not binary. Each retrieved chunk gets a relevance grade:

- `3`: contains the labeled gold evidence
- `2`: alternative but sufficient evidence
- `1`: relevant but incomplete
- `0`: irrelevant

The retriever score is average `NDCG@4` over 100 evaluation items.

Generator evaluation uses an LLM judge with three dimensions:

- correctness
- groundedness
- completeness

The notebook combines them as:

```text
0.5 * correctness + 0.4 * groundedness + 0.1 * completeness
```

It also reports latency, token estimates, and estimated OpenAI cost.

## How to run

Create an environment, install dependencies, set your API key, and run the notebook:

```bash
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

Set `.env` with your OpenAI key:

```bash
OPENAI_API_KEY=...
```

Then launch Jupyter:

```bash
jupyter notebook
```

Open [npr_mc_1.ipynb](/Users/stefanbinkert/Documents/FHNW_DS/NPR/NPR_MC_1/npr_mc_1/npr_mc_1.ipynb) and run it top to bottom.

## Implementation notes

- The notebook wraps the retrieval chain in `RAGChainWithSources` so answers include deterministic source section references.
- Section inference is manual and tailored to the naming conventions of the four manuals.
- The code saves the best vector index to `data/faiss_index/`.
- Qualitative analysis is built into the notebook and labels examples such as `Good retrieval and grounded answer` and `Alternative valid evidence retrieved`.

## Current limitations

- The project is implemented as one large notebook rather than reusable modules.
- There are no automated tests.
- Results depend on external APIs, model behavior, and local document availability.
- Some evaluation steps use an LLM judge, so scores are not fully deterministic.
